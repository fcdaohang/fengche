ICP 上的 Web 身份和身份验证#

原文链接 作者：Björn Tackmann，研究团队负责人 | DFINITY
Internet 计算机用更高级和更安全的密码验证方法代替了用户名和密码。

了解身份和身份验证在 Internet 计算机环境中的含义，世界上第一个以网络速度运行并且可以无限制地增加容量的区块链网络，我们必须首先考虑当今网络如何使用身份和身份验证。

当您登录网站时，您的用户名通常是您的电子邮件地址或一串字母和数字，它充当您的唯一标识符，将服务器上的相关数据与您的身份联系起来。您的密码是身份验证的手段。因为，理论上，只有你应该知道你的密码，服务器将您的密码解释为它正在与您通信的证据。但事实证明，密码实际上并不是一个好的远程身份验证机制。

当您在网站上输入密码时，您的计算机会将密码发送到服务器，并与密码数据库进行核对。不幸的是，黑客可以访问这些密码数据库。在最恶劣的情况下，密码以明文形式存储在服务器上，这不是推荐的安全做法。然而，即使密码被加密，恢复它们也只是黑客愿意投资以获得访问权限的计算和金钱资源的问题。

ICP 旨在通过跨多个数据中心复制数据和计算来提供安全性，以防止个别计算提供商的恶意行为。复制在保护数据完整性的同时，不能防止信息泄露；在 Internet 计算机上使用密码仍然会遇到与传统 Web 相同的安全问题。因此，对于 Internet 计算机，我们已将密码替换为适当的密码验证。

我们在 Internet 计算机上用于身份验证的主要加密机制是数字签名方案。数字签名是一个非常标准的概念。它们是在 1970 年代后期发明的，至少从 1990 年代中期开始就被广泛使用。

它们通常被认为由三种算法组成：

  • 密钥生成，您可以将其视为选择密码的模拟。密钥生成创建一对密钥：
  • 1）必须保密的私钥，就像密码一样
  • 2）从私钥派生的公钥，但可以公开。
  • 签名，它接受消息和私钥并生成签名。当我们使用数字签名进行用户身份验证时，该算法在持有私钥的用户端运行。
  • 验证，它获取消息、签名和公钥，并验证签名是否与消息和公钥匹配。这里的关键属性是，与检查密码不同，它需要将秘密密码存储在服务器上，这种情况下的签名验证可以仅基于公开信息进行。服务器存储一个公钥列表，每个用户一个，公钥和签名都不必保密。

互联网计算机上的应用程序是基于通过传递消息进行交互的容器智能合约实现的。更详细地说，交互模型基于请求，这很像远程过程调用。当容器 A 调用容器 B 时，容器 A 指定目标容器（即容器 B）、它所参与调用的函数的名称以及函数的参数。在容器 B 上评估指定函数时，该容器也知道该函数已由容器 A 调用。评估完成后，Canister A 获取函数的返回值作为响应。当用户与容器交互时，应用相同的远程过程调用模型。当用户调用容器时，用户会向目标容器发送请求。该请求还指定了一个带参数的函数，用户获得返回作为响应。在请求变化期间，容器还获知调用它的用户的身份。

上图是用户发送请求的示意图。中间的浅灰色区域显示了核心请求：目标容器 ID、函数名称、参数以及调用者的身份或主体。较深的灰色区域显示包含身份验证信息、签名和公钥。如左侧所示，调用者的主体是通过从公钥Hash得出的。该技术广泛用于区块链领域，例如比特币或以太坊地址。右侧显示了作为数字签名方案中的消息的请求内容如何通过签名绑定到公钥。当 Internet 计算机收到这样的请求时，它会检查签名在指定的公钥下是否有效，以及公钥和调用者主体之间的关系。

为了确保消息确实是由消息中指定的调用者发送的，容器不必费心处理这些技术细节。如果一切顺利，互联网计算机将评估容器上的指定功能。如果其中一项检查失败，则简单地丢弃请求。

我想提一些关于我们使用的主体格式的细节。我们从 DER 格式的公钥开始，并使用 SHA-224 对其进行哈希处理，得到一个 28 字节的字符串。我们附加一个字节，仅用于区分从公钥派生的主体与我们在 Internet 计算机中其他地方使用的主体，例如用于容器的主体。这 29 个字节是用户主体的内部二进制表示。在将主体转换为其文本表示时，我们首先添加一个 CRC-32 错误检测代码。然后，我们在 Base32 中对生成的字符串进行编码，最后构建每组 5 个字符，用破折号分隔。选择该格式是为了支持简单的复制粘贴和适当的错误检测，同时仍然允许 ASCII 表示少于 64 个字符以与 Internet 协议（如 DNS）兼容。

到目前为止，我们看到的方案在构造上仍然有点不灵活。它将用户的主体绑定到单个加密密钥。这种限制使用户难以与来自不同设备的容器进行交互，因为这需要在这些设备之间共享相同的加密密钥，从安全角度来看，这既乏味又令人沮丧。相反，我们使用不同加密密钥之间的委托。在上图中，您可以看到从黄色键到橙色键的委托。这样的委托由委托的密钥组成，橙色的密钥；一些附加参数，例如授权范围的到期或限制；和授权密钥的签名，黄色的。

当使用橙色密钥签署请求时，用户现在可以包含来自黄色密钥的委托，以便使用从黄色密钥派生的身份。代表团的特别强大之处在于它们可以组合。例如，橙色键可以将委托扩展到紫色键。如果从公钥基础设施和 X.509 中所有这些看起来都很熟悉，那绝非巧合——我们只是在使用更轻量级的数据结构。

委托的一种特定应用与 Web 身份验证有关。 Web 身份验证是万维网联盟 (W3C) 的最新标准，主要针对 Web 应用程序的双因素身份验证。该标准的动机是，如前所述，密码具有严重的安全缺陷。他们经常成为网络犯罪分子的牺牲品，无论是通过网络钓鱼电子邮件和安装在用户设备上的恶意软件，还是在服务被黑客入侵时。

双重身份验证意味着除了密码之外，登录 Web 应用程序还需要一个额外的安全因素，通常是用户拥有的安全设备。在实践中，这可能是一个安全的 USB 密钥或一个内置在用户终端设备中并通过生物识别激活的安全芯片。安全芯片存储密码密钥。由于加密密钥永远不会离开安全芯片，因此即使用户的计算机或手机感染了恶意软件，它们仍然是安全的。

当 Web 身份验证作为 Web 应用程序中的第二个因素时，协议流程如下： 用户通过提供用户名和密码发起登录过程后，Web 服务器将生成一个随机挑战并发送给用户的浏览器，然后浏览器将质询发送到安全设备，这需要用户交互才能签署质询。然后将签名的质询发送回服务器，服务器会根据用户注册的公钥验证质询上的签名。这确保了登录 Web 应用程序除了密码之外还需要持有安全设备。

我们正在建立这样一个事实，即 Web 身份验证是一种使用数字签名进行身份验证的开放标准，并且已经在广泛的设备上得到支持。然而，在将其用于 Internet 计算机时，我们必须克服一些障碍。Web 身份验证采用传统 Web 的面向会话的客户端服务器模型，其中用户在登录应用程序时进行一次身份验证，并在同一会话中发送后续消息。相比之下，Internet 计算机实现了一个模型，其中每个请求都单独进行身份验证。特别是，没有服务器可以生成要由安全设备签名的质询，因为在浏览器和 Internet 计算机之间没有状态会话。然而，回想一下，在典型的 Web 身份验证流程中，安全设备为服务器发送的质询提供数字签名。

为了使用相同的协议实现请求身份验证，我们使用请求本身作为挑战，并由安全设备对其进行签名，类似于我们的一般请求身份验证方案。我们必须克服的另一个问题是 Web 身份验证需要用户对每个签名进行交互。在互联网计算机服务的典型前端中，单个页面加载可以对应多个请求。由于我们不想要求用户明确确认每个请求，我们使用上述委托机制。当使用 Web 身份验证与容器交互时，我们首先生成一个短期会话密钥。然后，我们使用 Web 身份验证来签署针对该会话密钥的委托，以便单个用户交互可以触发对 Internet 计算机的多个请求。

虽然 Web 身份验证非常适合安全地存储加密密钥，但它不仅将这些密钥绑定到设备，还绑定到特定容器。这样做的原因是浏览器安全模型，它严格区分运行在同一 Web 浏览器中的不同应用程序可访问的状态的来源。在网络上，你可以认为一个来源大致对应于一个网站。在 Internet 计算机上，每个源对应一个容器。这种严格的状态分离对于安全性至关重要，但它也使诸如密钥备份或支持从多个设备无缝访问相同容器等功能变得乏味，因为所有这些操作都必须为每个容器单独执行。我们通过使用 Internet 身份服务来解决此问题，该身份提供程序类似于您在网络上熟悉的“使用 Google 或 Facebook 登录”功能。

当用户首次加载给定容器的前端时，该前端可以显示“使用 IC 登录”按钮。当用户单击该按钮时，浏览器会打开一个带有 Internet 身份服务的弹出窗口，这是一个允许用户管理密钥和身份的特定应用程序。在那里，用户可以决定是否允许容器前端使用用户的身份。如果用户同意，浏览器会被重定向到容器前端，并且可以以用户的身份访问容器。该机制再次使用会话密钥和委托机制。容器前端生成会话密钥对并将公钥传输到 Internet Identity。如果用户确认，Internet Identity 会生成一个委托，并将其返回给容器前端。作为通过大型技术提供商登录的额外好处，身份提供商中的完整身份验证流程发生在用户端，因此私人用户操作的暴露要少得多，因此跟踪也少得多。（互联网身份服务常见问题解答）

说到用户跟踪，互联网身份将为用户登录的每个容器前端提供不同的身份，这对安全和隐私非常有用。如果不是这样，互联网身份将允许每个前端在用户的单一主体下登录。如果该用户与不相关的服务（例如，留言板和购物网站）进行交互，这些可能会在背后关联用户在这些网站上的行为。更糟糕的是，留言板的前端可以恶意调用购物网站的容器，以用户的名义下单。因此，互联网身份服务为用户登录的每个前端生成不同的身份，前端由其主机名区分。这样，用户在不同服务上的操作就不会那么容易被跟踪。虽然前端仍然能够使用用户的身份调用 Internet 计算机上的任何容器，但只有与执行调用的前端相关联的身份。

我们最近发布的互联网计算机接口规范中详细说明了上述和互联网身份中使用的机制——我们期待着看到我们的开发者社区还将从中构建什么。

验证互联网身份代码：演练#

原文链接 作者：高级研究员兼工程师 Joachim Breitner | DFINITY

确认 ICP 区块链身份验证系统的确按照其所声称之方式运行的分步指南。

在互联网计算机上，用户可以使用互联网身份密码认证系统登录各种dapp，如NNS前端dapp、OpenChat等。在这样做的过程中，用户相信这项服务会妥善保管他们的凭据——但他们可能希望直接确认 Internet Identity 确实没有跟踪他们。

互联网身份是否真正运行它所声称运行的代码？为了帮助您回答这个问题，我将引导您完成验证这一点的步骤。（当然，以下内容也适用于其他容器，但在这种情况下，我将坚持使用 Internet Identity。）

找出正在运行的内容#

互联网计算机上的智能合约，即容器智能合约，是一个 WebAssembly 模块。Internet 计算机故意不允许您仅下载任何容器的 Wasm 代码，因为可能有些开发人员希望将其代码保密。但它确实暴露了 Wasm 模块的哈希值。最简单的方法是使用 dfx：

`$ dfx canister –no-wallet –network ic info rdmx6-jaaaa-aaaaa-aaadq-cai

Controller: r7inp-6aaaa-aaaaa-aaabq-cai

Module hash: 0xd4af9277f3e8d26fd8cdc7874a9f47b6456587fbb2a64d61b6b6880d144d3c04`

这里的“控制器”是治理容器的容器 id。这告诉您，互联网身份由网络神经系统 (NNS) 控制，其代码只能通过投票的提案进行更改。这很好；如果控制者只是我，我可以更改互联网身份代码并接管您的所有身份。

“模块哈希”是已部署的 .wasm 的 SHA-256 哈希。因此，让我们跟随那个踪迹。

找到正确的提交#

由于对 Internet 身份的升级是通过向 NNS 提出建议来完成的，因此我们应该在 https://github.com/ic-association/nns-proposals 存储库中的proposals/network_canister_management 目录中找到对此类建议的描述。

我们必须找到升级 Internet Identity 的最新提案。不幸的是，该文件夹包含许多容器的建议，并且文件命名不是很有帮助。我通常从底部浏览列表并查看第二列，其中包含创建或修改文件最新提交的标题。

在这种情况下，倒数第二个是我们关心的：该文件列出了基本原理，概述了更改，最重要的是，说 bd51eab 是我们要升级到的提交。

该文件还说 wasm 哈希是 d4a…c04，与我们在上面看到的相符。这很好：看来我们真的找到了升级互联网身份的最年轻的提案，并且提案实际上通过了。

警告：如果你是偏执狂，不要相信这个文件。没有什么可以阻止提案提出者创建一个指向一个修订的文件，同时在提案中实际包含不同的代码。这就是为什么接下来的步骤对于验证是必要的。

容器的ID是什么呢#

如果您仔细观察，您可能已经注意到我们获得了容器 rdmx6-jaaaa-aaaaa-aaadq-cai 的模块，但我们正在访问 https://identity.ic0.app/ 上的 Web 应用程序。那么这个联系在哪里呢？

将来，我希望 Internet 计算机上有某种形式的类似 DNS 的“好主机名注册表”，它存储从好名称到容器 id 的映射，并且您将能够查询“哪个容器服务于 rdmx6- jaaaa-aaaaa-aaadq-cai”以安全的方式（例如使用认证变量）。但是由于我们还没有这个，但仍然希望您能够为 Internet 身份使用一个好听的名称（并且以后不必更改名称，这会让人头疼），我们现在已经硬编码了这个映射.

此处的相关代码是您的浏览器在访问任何 * .ic0.app URL 时下载的“Certifying Service Worker”。然后，这段代码将拦截对该域名的所有请求，将其映射到查询调用，然后使用经过认证的变量来验证响应。事实上，映射在代码中：

const hostnameCanisterIdMap: Record<string, [string, string]> = { 'identity.ic0.app': ['rdmx6-jaaaa-aaaaa-aaadq-cai', 'ic0.app'], 'nns.ic0.app': ['qoctq-giaaa-aaaaa-aaaea-cai', 'ic0.app'], 'dscvr.ic0.app': ['h5aet-waaaa-aaaab-qaamq-cai', 'ic0.page'], };

其他容器呢？#

原则上，相同的方法适用于其他容器，无论是 OpenChat、NNS 容器等。但细节会有所不同，因为每个容器开发人员可能有自己的方式：

  • 传达其容器源的位置和修订版本
  • 建造容器

特别是，如果没有可复制的方式来构建容器，这将失败，这就是为什么像 https://reproducible-builds.org/ 这样的项目通常如此重要的原因。

互联网身份：用户名和密码的终结#

原文链接

Internet Identity 区块链身份验证系统使您能够安全且匿名地登录到 Internet 计算机上的 dapp。

互联网计算机是世界上第一个真正的互联网规模的公共区块链。与其他区块链不同，它可以处理和存储任意数量的数据。它还托管以网络速度运行的智能合约（区块链软件逻辑），效率比以前高得多。它们能够直接向最终用户提供交互式 Web 内容，使广泛的系统和服务能够以 dapp 的形式完全建立在区块链上，而不依赖于云服务等传统技术。现在终于可以完全在公共区块链上构建和运行超大规模的代币化社交媒体。

互联网计算机的目的是通过使用托管在网络空间中的智能合约，使几乎所有系统和服务都可以重建和重新构想为安全的 dapp，从而扩展互联网–它将处理和存储他们的所有数据，而无需传统技术及其服务的增强。这将推动“区块链奇点”，几乎每个系统和服务最终都完全从公共区块链运行。

为了避免困扰传统网络密码认证的安全问题，互联网计算机区块链引入了一种新的密码认证系统。这个被称为互联网身份的新系统非常易于使用，与用于通过网络对区块链进行身份验证的现有系统形成鲜明对比。

从本质上讲，它允许最终用户使用他们的设备对区块链上的 dapps 进行身份验证，这种方式通过互联网计算机使用的“链密钥”加密技术成为可能。最终用户设备可以是传统的 HSM 设备，例如 YubiKey，也可以是包含 TPM 芯片的移动设备，例如笔记本电脑或手机。

举例而言，成千上万的人已经在使用笔记本电脑上的指纹传感器或手机上的 FaceID 对完全在互联网计算机上运行的 dapp 进行身份验证。在每种情况下，都提供了出色的可用性，即使公钥加密技术被安全地应用在所涉及设备内部的安全芯片中，用户也不必自己接触密钥材料，并从无缝、低摩擦的体验中受益。

Internet Identity 正在不断改进，以使其与越来越多的支持称为 WebAuthn 的 Web 标准的设备兼容。如果您的设备较旧，并且您的浏览器不支持适用于 Internet 身份的标准（以便您可以使用例如指纹传感器进行身份验证），然后您可以使用诸如 YubiKey 之类的 HSM 设备来增强它。这也可以很方便。

例如，您可以购买支持 NFC 的 YubiKey，它允许您通过手机等方式登录，只需用钥匙轻敲即可。幸运的是，消费者设备越来越多地默认包含此类身份验证器。

互联网身份使用户能够通过他们的设备安全地对区块链上的 dapp 进行身份验证的方式是革命性的。

您可以直接在 https://identity.ic0.app/ 尝试互联网身份认证配置器（该页面直接来自互联网计算机区块链）。

这使您可以创建称为 IIs（发音为 eye-eyes）的身份。每个身份都有一个“身份锚”，您可以记住它以重新连接（身份锚刻意较短）。这些锚不是以任何方式解锁 II 的秘密，但你不应该分享它们。

您可以创建任意数量的 II。例如，您可以为区块链社交媒体 dapps 创建一个 II，为 DeFi 和 Staking 创建另一个 II，等等。您可以为每个 II 添加任意数量的设备。例如，如果您创建一个用于社交媒体的 II，您可能会添加笔记本电脑的指纹传感器、手机上的 Face ID 以及一个或两个 YubiKey 用于备份（因此，如果您丢失了笔记本电脑和手机，您可以重新连接 II，删除旧设备并添加新设备）。

当您对区块链上的 dapp 或服务进行身份验证时，它会看到一个专用的假名而不是电话号码，这可以防止您在 dapp 和服务中被跟踪，并保持您的匿名性。

想象一个世界，您可以：

  1. 无需电子邮件、用户名或密码即可安全地在线验证自己的身份 - 只需使用您的设备即可登录。
  2. 登录互联网服务而不会被跟踪，也不会被科技公司挖掘您的信息。
  3. 与您今天使用的几乎任何类型的身份验证系统相比，您可以更方便地进行身份验证。

互联网身份的工作原理#

Internet Identity 建立在 WebAuthn 协议之上，并使用安全的加密身份验证，为用户提供了三种身份验证选项：

  1. 智能手机或笔记本电脑中的内置生物特征验证方法（例如，Face ID、Touch ID 或指纹扫描仪）。
  2. 用户通常用于解锁其计算机或手机的密码或 PIN。
  3. 插入计算机 USB 端口的安全密钥（例如 YubiKey）。

当您首次通过区块链网址 https://identity.ic0.app 上的交互式网页内容创建互联网身份时，您设备上的安全芯片将生成一个唯一的加密密钥，其公钥将存储在互联网上计算机以及为您生成的身份锚。

Identity Anchor 是一个总括身份，您可以在该身份下重新连接到您希望使用的各种设备上的 II。您可以使用与 II 关联的身份锚来注册新设备，这样您就可以在所有设备上无缝使用应用程序。

仔细记下与您使用的每个 II 相关联的身份锚。如果您丢失了身份锚，您将无法使用 Internet 身份来管理您的设备或访问您的应用程序。此外，在同一身份锚下添加多个身份验证设备以实现冗余至关重要。

Identity Anchor 的一个主要优点是它对安全性不敏感。它不会与任何 PIID 绑定，因此是否有人了解身份锚并不重要（受上述问题的影响）。

用户如何使用 Internet Identity 与 dapp 进行交互？#

当用户首次加载给定容器智能合约的前端时（例如，当使用 dapp 或“开放互联网服务”，这是由代币化治理系统控制的自治 dapp）时，该前端会显示一个按钮供用户使用验证自己，类似于许多人已经熟悉的单点登录 (“SSO”) 服务。

当用户单击该按钮时，浏览器会打开一个带有 Internet 身份的弹出窗口，允许用户管理密钥和身份。

使用互联网身份，用户使用他们选择的设备和方法进行身份验证，然后授权访问应用程序。然后，浏览器重定向到canister智能合约前端，可以访问用户选择的II下的canister。 （此机制使用会话密钥和委托机制。容器前端生成会话密钥对并将公钥传输到 Internet 身份。如果用户确认，互联网身份会生成一个委托并将其返回给容器前端。）

一旦您使用已经概述的三种方法之一在浏览器中验证了您的身份，系统将提示您确认您的注册。要注册您的设备，您将创建一个身份锚。这个身份锚是独一无二的，但它不是秘密，所以你应该把它保存在多个地方。您的浏览器会记住您使用的 II 的身份锚，但如果您在另一台计算机上登录或清除浏览器状态，您将需要它。同样，如果您丢失了身份锚并在所有设备上注销，您将无法使用相关的 II。

与通过单点登录进行登录相比，身份提供者中的完整身份验证流程发生在用户端，因此私人用户行为的暴露要少得多，大型科技公司跟踪的风险也较小。

值得注意的是，互联网身份还将为用户登录的每个容器前端提供不同的假名身份，以防止他们被跟踪，从而增强匿名性和隐私性。

作为一个思想实验，如果 Internet 身份没有为用户登录的每个容器前端提供不同的身份，那么 Internet 身份将允许每个前端在用户的单一主体下登录。如果该用户与不相关的服务交互——例如，社交媒体网络和电子商务网站——这些不相关的公司可以关联用户在这些网站上的行为。在最坏的情况下，社交媒体网络的前端现在可以恶意调用电子商务网站的罐子并冒充用户下订单。

这就是为什么互联网身份为用户使用某些特定 II 进行交互的每个 dapp 和服务提供不同的假名身份的原因。

改进的身份验证#

当您访问使用身份验证系统的去中心化应用程序 (dapps) 时，互联网身份使您能够安全且匿名地进行身份验证。

与大多数身份验证服务不同，您的 Internet 身份不需要您设置和管理密码、生成加密安全种子短语或向应用程序或服务提供任何个人识别信息。本质上，在幕后，Chain Key 加密用于验证您与 II 关联的每个设备。现在您可以使用您选择的身份验证方法，例如智能手机的面部识别、您的计算机解锁密码或安全密钥。
