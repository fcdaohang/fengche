互联网计算机——世界上第一个网络速度、互联网规模的公共区块链——全面推出之前的最后一个里程碑即将到来。

在2020 年 9 月 30 日发布 Sodium 的虚拟活动中，DFINITY 基金会将推出网络神经系统NNS，这是一个控制互联网计算机开放算法的治理系统。该活动还将提供有关高级密码学、共识协议和代币经济学的深入技术资料。

在这个重要时刻之前，我们希望向公众提供非常高级的有关网络如何运作的概述。

“极客的互联网计算机”：新的 DFINITY 白皮书

网络神经系统 Network Nervous System#

互联网计算机基于称为互联网计算机协议（ICP）的区块链计算机协议。网络本身是由具有层次结构的区块链构成的。底部是托管专用硬件节点的独立数据中心。这些节点机器组合起来创建子网。子网托管容器 canister 智能合约，它们是由用户上传的可互操作的计算单元，包含代码和状态。

使 ICP 独一无二的元素之一是网络神经系统 (NNS)，它负责控制、配置和管理网络。数据中心通过向负责引入数据中心的NNS申请加入网络。虽然 NNS 本身有一个开放的治理系统，但它监督参与网络的权限。从某种意义上说，它在互联网上扮演着相当于 ICANN 的角色，除其他外，它为那些想要运行 BGP 路由器的人分配自治系统编号。NNS 履行广泛的网络管理角色，包括监视节点机器以查找 Internet 计算机网络上的统计偏差，这可能表明性能不佳或错误行为。

NNS 在网络的代币经济中也发挥着关键作用。 NNS 生成新的 ICP 代币（以前称为 DFN 代币）来奖励由数据中心运行的节点和在 NNS 内投票的神经元，这是它决定提交给它的提案的方式。当 NNS 创建新的 ICP 代币来奖励数据中心和神经元时，它是通货膨胀的。

最终，数据中心所有者和神经元所有者可以拿走他们的代币并与容器所有者和管理者交换它们。容器所有者和管理者获取这些代币并将它们转换为循环，并使用这些周期为他们的容器充电。例如，当这些容器执行计算或存储内存时，它们会在循环中燃烧，最终它们必须通过更多循环充电才能继续运行。那是通货紧缩。

子网#

要了解 Internet 计算机，您必须了解子网的概念，它是整个网络的基本组成部分。子网负责托管由 Internet 计算机网络托管的软件容器的不同子集。通过以 NNS 控制的方式将来自不同数据中心的节点机器聚集在一起来创建子网。这些节点机器通过 ICP 进行协作，以便对称地复制与它们托管的软件容器有关的数据和计算。

NNS 在构建子网时结合了来自独立数据中心的节点。这使得 ICP 协议数学能够使用由 DFINITY 开发的拜占庭容错技术和密码学来确保子网是防篡改和不可阻挡的。尽管子网是整个 Internet 计算机网络的基本组成部分，但它们对用户和软件是透明的。用户和容器软件只需要知道容器的身份就可以调用它共享的功能。

这种透明度是互联网基本设计原则的延伸。在互联网上，如果用户想连接一些软件，他们只需要知道运行该软件的机器的 IP 地址和该软件正在侦听的 TCP 端口即可。在互联网计算机上，如果用户想要调用一个函数，他们只需要知道容器的身份和函数签名。与互联网创建无缝连接的方式相同，DFINITY 为软件创建了一个无缝世界，任何获得许可的软件都可以直接调用任何其他软件，而无需了解网络的底层运作。

互联网计算机还以其他方式确保子网的透明性。 NNS 可以拆分和合并子网，例如，为了平衡整个网络的负载。这对于托管的容器也是透明的。

在此示例中，我们有一个虚拟子网 ABC，它托管 11 个容器。 NNS 告诉它分裂。子网 ABC 继续使用容器 1-6，并生成一个新子网 Subnet XYZ，该子网继续使用容器 7-11。所涉及的容器都不会出现服务中断。

当您将容器上传到 Internet 计算机时，您必须以特定子网类型为目标。实际上有一个托管 NNS 的特殊子网，但您不能将您的容器上传到该子网。相反，您必须针对子网类型，例如“数据”、“系统”或“信托”。

每种子网类型都为您的容器提供了某些属性和功能。例如，如果您的容器托管在数据子网上，它可以处理调用，但不能调用其他容器。为此，您需要一个系统子网。如果您希望您的容器能够保持 ICP 代币的余额或将周期发送到其他容器，您将需要一个受托子网。由于这些原因，治理容器只能托管在受托子网上。

子网的能力部分来源于底层的容错。这是我们希望尽快与公众分享的基础科学中一个非常令人兴奋的领域，包括允许 NNS 修复损坏的子网的新密码学。

Canisters 容器#

子网的目的是托管容器。容器在专用管理程序中运行，并通过公开指定的 API 相互交互。容器内部是可以在 WebAssembly 虚拟机上运行的 WebAssembly 字节码以及它在其中运行的内存页面。通常，该 WebAssembly 字节码将通过编译一种编程语言（例如 Rust 或 Motoko）来创建。该字节码将包含一个运行时，使开发人员可以轻松地与 API 交互。

在 Internet 计算机上，容器共享的功能必须以两种方式之一调用。它们可以作为更新调用或查询调用调用。本质区别在于，当您将函数作为更新调用调用时，它对容器内存中的数据所做的任何更改都将保留，而如果将函数作为查询调用调用，则它对内存所做的任何更改都将被丢弃运行后。

更新调用会进行持久更改，而且它们也是防篡改的，因为 ICP 区块链计算机协议在子网中的每个节点上运行它们。如您所料，调用在一致的全局调用顺序中运行，使用允许在完全确定的执行环境中并发执行的机制。更新呼叫只需两秒钟即可完成。

在此示例中，用户向容器内托管的金融交易所提交购买订单。

另一方面，查询调用不会持久化更改。它们对内存所做的任何更改在运行后都会被丢弃。它们非常高效且价格低廉，只需几毫秒即可完成。这是因为它们不会在子网中的所有节点上运行，这也意味着它们提供的安全级别较低。

正交持久性#

互联网计算机最有趣的事情之一是开发人员保存数据的方式。开发人员不必考虑持久性——他们只需编写代码，持久性就会自动发生。这称为正交持久性。那是因为互联网计算机保留了运行代码的内存页面。

您可能想知道这一切是如何运作的。关于可以改变内存页面的更新调用，容器是软件参与者。这意味着在任何给定时间，容器内只能有一个执行线程。

尽管容器内只有一个执行线程，但默认情况下可以交错跨容器更新调用。当更新调用进行跨容器更新调用时会发生这种情况，这会阻塞，从而允许将执行线程移动到新的更新调用。

相比之下，查询调用不会对内存进行持久更改。这允许在任何给定时间在容器内有任意数量的并发线程处理查询调用。这些查询调用针对记录在最后一个最终状态根中的内存快照运行。

最后，如果不提到容器可以创建新的容器，并且容器可以自己分叉，那么对容器的讨论是不完整的。你可以简单地通过指定 WebAssembly 字节码来创建一个新的容器，并且内存页面一开始是空的。当容器分叉自己时，会创建一个新生成的副本，该副本与内部的内存页面完全相同。在创建可扩展的互联网服务时，分叉被证明非常强大。

可扩展性#

现在是对扩展的互联网服务的高级解释。容器的各种容量都有上限。例如，由于 WebAssembly 实现的限制，一个容器只能存储 4GB 的内存页面。出于这个原因，当我们想要创建扩展到数十亿用户的互联网服务时，我们必须使用多容器架构。

我们可能希望创建一些特殊的容器就足够了，创建容器的大量副本，然后将用户内容分片到不同的容器，以创建可以横向扩展的互联网服务。不幸的是，由于多种原因，这种架构过于简单。

确实，每个额外的容器都会增加整体内存容量。每个额外的容器都会增加整体更新和查询调用吞吐量，这也是事实。但是我们无法扩展针对特定用户内容的查询调用请求。每当我们通过添加更多容器分片来增加系统容量时，我们还需要重新平衡用户内容，这并不是一个真正好的边缘架构。也没有明显的方法可以从靠近他们的副本向最终用户提供查询调用。我们将需要前端容器和后端容器。

Internet 计算机为将最终用户连接到前端容器提供了一些有趣的功能。其中之一允许通过 NNS 将域名映射到多个前端容器。当最终用户希望解析这样的域名时，Internet 计算机会查看托管前端容器的所有子网中的副本节点的总数，并返回最接近的副本节点的 IP 地址。这导致最终用户在附近的副本上执行查询调用，从而减少了固有的网络延迟并改善了用户体验，从而在没有内容分发网络的情况下提供了边缘计算的好处。

为了充分利用这个功能，我们需要一个经典的架构，包括前端容器和后端数据桶容器。在此示例中，Web 浏览器希望加载个人资料图片。

首先，浏览器将映射到一个前端容器，该容器在具有附近节点的子网上运行。然后，浏览器将提交查询调用请求以将照片检索到附近的节点。 前端容器随后将向保存照片的数据桶容器发出跨容器查询调用请求。 如果数据桶容器返回的查询调用响应涉及静态内容，例如照片，则可以将数据存储在缓存中。在这种情况下，运行前端容器查询调用的副本节点可以将查询调用响应输入到其查询缓存中。 当然，查询调用缓存机制对前端容器代码是完全透明的。一旦用户调用的前端容器收集了所有必要的信息，它就可以通过查询调用响应或通过 HTTP 端点返回内容。 随着时间的推移，节点的查询缓存会积累静态内容并生成附近用户感兴趣的数据，从而为他们提供更快、更好的用户体验。通过这种方式，互联网计算机的本地边缘架构提供了内容分发网络的好处，但开发人员无需做任何特别的事情，也无需寻求单独的专有服务的帮助。 对于更新调用，经典架构采用不同的方法。有必要对用户的内容和数据进行序列化更新，以防止更新丢失等问题。通常，这是通过例如仅通过散列用户名将用户映射到特定前端容器来实现的。 一旦在 Web 浏览器或智能手机上运行的 UX/UI 确定了哪个前端容器负责协调对某些内容或数据的更改，它就可以通过向其标准接口提交更新调用来修改该内容或数据。 然后，此前端容器通常会进行更多跨容器更新调用以实现所需的更改。

开放式互联网服务#

为了总结这一切，让我们讨论使用我们的两级架构设计一个开放的互联网服务，包括前端容器和后端数据桶容器。首先，当您编写前端容器代码时，您将通过使用名为 BigMap 的现有库类来简化您的工作。 BigMap 可以存储 EB 级的数据，您只需使用一行代码就可以向其写入对象。这种架构将通过前端容器和数据桶容器分叉来透明且动态地横向扩展，以便在两个容器之间划分分配给一个容器的对象的责任。

最后，为了创建一个真正的开放互联网服务，您将把所有容器的责任分配给一个开放的代币化治理容器。如果您是一名企业家，您将通过在早期出售其中一些治理代币来筹集开发资金。你可能会设计方案，通过给予他们治理代币来激励你的互联网服务的早期参与者，以获得更好的网络效应——并获胜。
