原文链接

Motoko 是一种新的智能合约编程语言。它被设计用于无缝支持IC的编程模型，以更容易利用区块链的独特功能。 Motoko 是强类型的、基于参与者的，并且内置了对正交持久性和异步消息传递的支持。生产力和安全功能包括自动内存管理、泛型、类型推断、模式匹配以及任意精度和固定精度的算术。消息传递透明地使用IC的 Candid 接口定义语言和有线格式，以实现类型化、高级和跨语言互操作性。

前言#

DFINITY 基金会的 Motoko 团队很高兴地宣布开源 Motoko 编译器、解释器、测试套件和文档。

经过三年的开发和一年半以来的公开介绍，Motoko 的完整源代码现在可以在 Apache 2.0 许可下使用（连同其以前开源的基础库）。

Motoko 是一种编程语言，旨在无缝支持IC的编程模型，从而利用该平台的一些更独特的功能并更高效容易地构建应用程序。Motoko 是强类型的、基于参与者的，并且内置了对正交持久性和异步消息传递的支持。生产力和安全功能包括自动内存管理、泛型、类型推断、模式匹配以及任意精度和固定精度的算术。消息传递透明地使用 Internet 计算机的 Candid 接口定义语言和有线格式，以实现类型化、高级和跨语言互操作性。

我们希望此代码版本能够促进与更广泛社区的协作和贡献，无论它们涉及改进文档、完善错误消息，还是生成全新的工具，例如额外的 IDE 集成、调试器支持和代码格式化工具。

我们的目的是为内部和外部贡献者提供相同的开发体验。目前，我们的测试基础设施仍部分依赖于内部服务，但我们正在努力将其替换为可公开访问的服务。外部和内部开发人员将处于平等地位。毫无疑问，当我们进入公开市场时，我们会遇到一些成长的痛苦。请多多包涵。

对于那些还不熟悉 Motoko 的朋友，这里将回顾并介绍它是什么、我们为什么开发它以及它是如何工作的……

WebAssembly#

要了解 Motoko，我们首先得简单谈谈 WebAssembly — 又名 Wasm（是的，拼写正确，没有全部大写）。您可能知道，Wasm 是一种新的低级代码格式，旨在实现可移植、安全和高效。

它最初的用例是 Web，但这个名字实际上是用词不当：当我们在 W3C 工作组中设计 Wasm 时，我们小心翼翼地将其作为一个开放标准和一个通用平台来做。也就是说，它不针对任何特定的编程语言、范式、计算环境或平台，并且我们确保它与网络完全无关。因此，Wasm 在许多其他环境中得到采用绝对不是偶然的，例如云计算、边缘计算、移动、嵌入式系统、物联网和区块链。

Wasm 有很多很多的设计考虑，有些是明显的，有些是相当微妙的。太多了，无法全部涵盖。关于 Wasm 的技术目标、设计选择、形式语义和实现技术的相当全面的讨论可以在我们发表在 Communications of the ACM 上的一篇科学文章中找到（本文的旧版本和技术含量更高的版本可在此处免费访问）。

Wasm 与其他虚拟机相比的主要区别在于它没有针对任何特定的编程语言进行优化，而只是抽象了底层硬件，字节码直接对应现代 CPU 的指令和内存模型。最重要的是，Wasm 通过强大的模块化和严格的数学规范支持沙盒，确保执行安全，没有未定义的行为，并且（几乎）完全是确定性的。而且，这些属性实际上有机器验证的数学证明！

总而言之，这些属性旨在使 Wasm 成为对可移植性、安全性、通用性和性能有很高期望的各种环境和用例的有吸引力的选择——例如IC。

Wasm 的属性使其成为表示在 IC上运行的程序的明显选择。但在实践中，将现有的编程语言移植到 Wasm 并非易事。显然，它需要实现一个新的后端编译器。这很有趣，但工作并没有就此结束：它还需要移植语言的运行时系统和库原语。还有一些功能，特别是与更高级语言相关的功能，目前无法在 Wasm 中轻松实现— 例如：线程、协程、异常和尾调用。虽然各种通过各自功能丰富 Wasm 的提案即将出现，但它们尚未最终确定以进行标准化。

尽管已经有许多针对 Wasm 的实验性语言实现，但大多数还没有准备好迎接黄金时间。主要包括低级系统语言，如 C/C++ 和 Rust。这些对于它们的用例来说当然很棒，但是对于为 Internet 计算机开发高级应用程序来说，它们并不是理想的工具，与手动干预内存管理相比，可访问性、生产力和高可靠性往往更受欢迎。

同时，互联网计算机的语言需要提供对平台主要概念的访问：具有异步消息传递的分布式编程模型，循环（又名gas）等资源的概念，以及其他一些特质。当然，它们都可以作为库提供，但是原生包含适当结构的语言可以提供更加无缝的编程体验。

因此，如果我们无论如何都必须开展工作才能起步，为什么不将其应用于创造能够提供最佳用户体验并传达我们对如何对互联网计算机进行编程的愿景的东西呢？

Motoko#

这就是为什么——尽管创建另一种语言存在所有风险——我们还是决定创建 Motoko。我们想要一种安全、易于使用、无缝公开平台概念的语言，以及一种对大多数程序员来说足够友好和易于使用的语言。目前，后一个目标使它几乎不可避免地处于分号和花括号的语言阵营中。而且这个营地里没有合适的语言。

但 Motoko 相当传统的皮肤只是表面的：它的内部是现代语言的结构。例如，每个构造都是一个表达式，它有闭包，它有变体类型和静态检查模式匹配，它有垃圾收集，当然它有一个灵活的类型系统，它实际上是健全的，即它确实保证了不存在某些错误，例如崩溃、未定义的行为、误解数据或只是在 switch 中遗漏了一个案例。没有漏洞！

同时，我们有意不去幻想或重新发明轮子，而是建立在丰富的实践和理论历史的基础上，并承认该领域数十年来的经验教训。除了将易于理解的功能组合在一起之外，Motoko 的设计还包含许多小决策，以最大限度地减少脚炮和安全方面的错误，例如，默认情况下数字不会溢出，默认情况下局部变量是不可变的，默认情况下并发执行是原子的, null 默认不能出现，字段默认是私有的，以此类推。哦，没有继承，只有子类型。

实现 Motoko 的这些部分并将它们编译为 Wasm 是传统的编译器工艺。用 OCaml 编写的 Motoko 编译器使用类型化的中间表示，一些转换传递，并输出 Wasm 字节码。生成的 Wasm 模块包括一个用 C 和 Rust 编写的小型运行时系统，主要实现一个使用 Wasm 内存作为其堆的简单垃圾收集器。这并不难，但肯定有很大的改进潜力，我们正在努力。

Actors#

然而，Motoko 的核心特性是它对 Actors 的直接支持，包括语法和类型系统。Actor 模型是一个众所周知的概念，已有 40 多年的历史，但遗憾的是，它几乎没有成为主流语言。 Actor 就像一个对象（在 Motoko 中，甚至看起来像一个对象），因为它封装了私有状态以及一组方法来处理可以发送给它的消息。但是所有消息发送都是异步的。因此，与 OO 中的传统方法不同，actor 方法没有结果。此外，所有的消息都是由一个actor顺序接收的——也就是说，它有一个隐式的消息队列，方法是原子执行的，即使消息是并发发送的。

Actor 是并发编程的绝佳模型，因为它们自动防止竞争条件（由于原子性和封装状态）和死锁（因为执行从不阻塞），因此排除了许多并发错误。所有这一切都不需要程序员定义一个锁。 Actors 也是分布式编程的一个很好的模型，因为异步自然地处理了将消息发送到潜在的远程接收器所涉及的延迟。最后，actor 非常适合 Dfinity 的IC，其中应用程序以所谓的罐的形式部署——本质上，actor 表示为可以跨子网通信的 Wasm 模块。因此，Motoko actor 编译为 Wasm 模块，其中方法成为导出的 Wasm 函数，具有平台定义的特殊约定。

简而言之，Motoko 中的应用程序是一个（或多个）actor，而后者又是一个编译成 Wasm 模块的大型异步对象。使用 Wasm 的内存概念，这样的参与者可以立即管理多达 4 GiB 的内部状态，尽管这可以通过链接多个 Wasm 模块来进一步扩大，每个模块都有自己的内存。

Futures#

为了使异步编程更方便并允许以顺序的“直接风格”表达它，Motoko 采用了另一个 40 多年前的编程语言研究史中的想法，尽管幸运的是最近变得有点流行：futures（也称为一些社区的承诺）。在 Motoko 中，它们以“异步值”的形式实现，类型为 async<T> 的值由带有 async 关键字前缀的表达式生成。特别是，函数体可以是异步表达式，从而自然地替换和概括存在于某些其他语言中的“异步函数”这个更单一的概念。

有了这个，actor 方法毕竟是允许有结果的——只要那些是 futures。它们可以等待来获得它们的价值，但只能在另一个异步表达式中，类似于其他现代语言中已知的 async/await monad。

Motoko 编译器通过传统的 CPS（延续传递风格）转换来实现这一点，将每个等待点转换为一个单独的 Wasm 函数（加上一些闭包环境），表示等待的延续。事实上，它是双管 CPS，因为每条消息也可以有一个失败回复和相应的失败延续。按照惯例，具有异步结果的方法是发送带有结果值作为参数的回复消息的方法。此消息由创建的延续函数接收，然后可以恢复它已捕获的执行。等待回复不会阻止参与者——同时它可以自由地接收其他消息。

持久性#

Motoko 的另一个重要考虑因素是允许开发人员利用区块链技术，而无需学习全新的计算类型。因此，我们提取了您可能需要的有关当前区块链编程语言品种的大部分特殊知识。例如，没有可观察的块或块高度的概念，没有用于更新区块链状态的显式结构，也没有其他 API 用于将数据写入持久存储，如文件或数据库（尽管可以将其模拟为库） .取而代之的是，互联网计算机实现了正交持久性——这是另一个古老的想法，即程序具有“永远”运行的错觉，并且它的内存只是保持活动状态（至少在它被明确删除之前）。在 Motoko 中，这意味着开发人员不必担心在消息之间显式保存数据，也不必担心文件或外部数据库：当下一条消息到达时，存储在程序变量中的任何值或数据结构仍然存在，即使那是几个月后。

超越 Motoko：Candid 的接口定义#

因为互联网计算机运行 Wasm，所以 Motoko 只是创建应用程序的一种选择——而且是有意的。我们还提供 Rust，我们期待提供其他语言选择。即使这样，由于每种语言都会统一编译为以 Wasm 表示的容器，因此这些容器可以通过消息发送自由地相互通信，而不管它们的源语言如何。

为了使这种互操作性得到良好定义，我们引入了一种名为 Candid 的通用接口定义语言 (IDL)。它是互联网计算机上的通信通用语，完全独立于元子。它描述了容器可以理解的一组消息以及一起发送的数据类型。在 Candid 中，数据由独立于 Motoko 类型系统或任何其他编程语言的规范数据类型（数字、文本、数组、记录、变体、函数、对其他容器的引用）的组合来描述。

呸，又一个类型系统？好吧，程序员可能会很高兴 Motoko 编译器可以自动使用和生成这样的 actor 导出和导入接口描述，并将它们从相应的 Motoko 类型映射到相应的 Motoko 类型。它还自动生成正确的 Wasm 代码来序列化和反序列化每条消息的参数数据，透明地将 Motoko 的内部表示与 Candid 指定的二进制格式相互转换。

通过这种方式，Motoko 程序可以以类型化的方式与外部容器通信，并表达远程调用，就好像它们是程序中的本地对象一样。这与远程容器是用 Motoko 还是用 Rust 编写的无关；一个容器的接口描述作为类型信息就足够了。除了方便之外，接口还提供了一种强大的模块化形式，程序可以针对其他参与者/容器进行类型检查，而无需访问它们的实现。

总结#

我们的目标是让所有语言在 Internet 计算机上拥有平等的权利，所有语言都在 Wasm 中编译到容器中，并且所有语言都通过 Candid 进行无缝通信。这对于打开 Internet 计算机很重要。 Motoko 只是众多选择中的一种，但它旨在成为一个特别适合在 Internet 计算机上开发的广泛应用程序的选择。
